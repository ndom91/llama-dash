llama-dash · admin polling

ETags make our existing dashboard polling cheap when nothing changed.

Before this change, every 5-second poll downloaded and parsed full JSON even when the dashboard data was identical. Now the server fingerprints JSON responses, the browser asks "same as last time?", and unchanged polls come back as tiny 304 Not Modified responses.

server: src/server/admin/handler.ts client: src/lib/api.ts benefits existing React Query polling
what changed headers Same routes, same schemas, same polling cadence. The contract gained ETag and If-None-Match.
what shrinks body bytes Unchanged responses send no JSON body, so repeated dashboard polls stop moving duplicate payloads.
what stays freshness Routes still run and compare the latest body, so the UI updates as soon as the payload changes.

Where the mechanism lives

server
GET /api/system
  route builds JSON
  admin dispatcher computes weak ETag
  if If-None-Match matches:
    return 304 + ETag + Cache-Control
  else:
    return 200 + JSON + ETag
client
getJson(url, schema)
  cached = jsonCache.get(url)
  fetch(url, { If-None-Match: cached.etag })
  if 304:
    return parse(schema, cached.data)
  if 200:
    cache { etag, data }
    return parse(schema, data)

Plain-English summary

We kept polling because operational dashboards need fresh state. ETags make that polling less wasteful: if the server's latest JSON is byte-for-byte equivalent to what the client already has, the server says "not modified" and the client reuses its validated cached object. The UI stays just as fresh, but quiet periods stop sending the same JSON over and over.

Why weak ETags?

The implementation uses weak ETags because this is a semantic cache validator for JSON API payloads, not a byte serving mechanism for files or ranges. We only need to know whether the response representation is unchanged enough for the dashboard to reuse the previous parsed value.

Why put it in the admin dispatcher?

Most admin routes already return through the shared json() helper and dispatcher. Adding the comparison at the dispatcher means existing and future JSON GET routes benefit without adding duplicate ETag logic to every handler.

Why still parse cached data through Valibot on 304?

The cached value came from a previous validated response, but parsing it again keeps the fetch helper's contract simple: callers always receive schema-validated data, regardless of whether the network response was 200 or 304.
